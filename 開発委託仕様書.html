1. 本件の目的と背景

本件は、GREEN×EXPO 2027（横浜花博）大阪府ブースで運用する来場者参加型のWeb体験システムの本番開発を委託するものです。来場者がスマートフォンで庭園内を巡り、最後に「花」を選んで送信すると、ブース内の大型サイネージに映る「花が咲く街」へその花が加わっていく演出を実現します。

📌 重要：プロトタイプは完成済みです 動作するプロトタイプ（表示ページ・API・モバイルページ一式）がすでに存在します。本仕様書は「プロトタイプを会期中の実運用に耐える本番システムへ仕上げる」ことを委託範囲とします。ゼロからの新規開発ではなく、既存実装の引き継ぎ・改修・本番化を前提とします。プロトタイプのソースコード一式と動作URLは別途共有します。

3. 開発スコープ（本番化対象）

プロトタイプは機能的に一通り完成しています。本委託の中心は、以下の本番運用に必要な品質・信頼性の作り込みです。詳細は第9章に記載します。

項目優先度概要
データ永続化基盤の本番化必須現状は無料ホスティングのため再起動でデータが消える。永続DBへ移行。
本番ホスティング選定・構築必須会期中（数週間）の安定稼働に耐えるサーバー環境。
メッセージ投稿の監視（モデレーション）必須公開表示前の不適切投稿対策。NGワードフィルタの強化と運用フロー。
CORS・セキュリティ設定の本番化必須現状は全許可。本番URLへの限定と基本的な堅牢化。
OTOMOサイネージ実機での表示検証必須実機の表示仕様（リロード方式・周期）に合わせた調整。
負荷・同時アクセス対策推奨想定来場規模に応じたレート制御・スパム対策。
運用監視・障害時復旧推奨無人運用前提の死活監視・自動再起動・ログ。
管理画面（投稿確認・統計）任意運用者向けの簡易ダッシュボード。

4. ① 表示ページ（サイネージ）要件

項目要件
表示環境OTOMO サイネージ（43インチ縦型を想定）のWeb表示機能。フルスクリーンのChromeベースを想定。必須
レイアウト縦長基準。画面下部に街のスカイライン、上方に空。花は街並みの隙間や空へ分散配置（重なりすぎない）。
更新方式一定間隔（sinceパラメータで差分取得）でAPIをポーリング。WebSocket等のリアルタイム双方向通信は使わない。ポーリング間隔はOTOMO側の読込周期に合わせ設定変更可能にする。
一斉開花演出読込時に受け取った新規分を、0.1〜0.3秒ずつずらして「ふわっと咲く」（透明度＋拡大）。「数分ごとにみんなの花が一斉に咲く」リズムある演出。
カウンタ表示「本日 ◯◯ 輪 / 累計 ◯◯ 輪」を常時表示。
障害耐性屋外・無人運用前提。ネットワーク断・APIエラー時もエラー画面を全面表示せず、前回表示を保持し静かに再試行。完全リロードで状態がリセットされても「現在の全花を描く」フォールバックで破綻しない。必須
デバッグ機能キー操作等で花を手動追加できる確認機能。本番ではフラグで無効化。
⚠ OTOMO実機の仕様が表示設計を左右します Web表示が「完全リロード」か、読込間隔が固定か可変か、リロード時のちらつき有無 ―― これらは提供元（ファンテック社）への確認事項です。実機検証を経て最終調整が必要です。

5. ② サーバー／API 要件

項目要件
役割花データの登録・永続保存・集計と、表示ページへの差分提供。
入力バリデーションcolor / kind は許可リストで検証（不正は422）。message は最大文字数制限とNGワードフィルタ。
モデレーション公開前提のため、不適切メッセージ対策が必須。NGワードフィルタの強化に加え、本番では「管理者が確認してから表示」または「自動フィルタ＋事後非表示」等の運用方式を設計・実装する。必須
個人情報個人情報は一切収集しない設計。氏名・連絡先・位置情報等を保存しない。
CORS本番では表示／モバイルの公開URLに限定する。必須

6. ③ モバイルページ要件

項目要件
動作前提アプリのインストール不要。QRから開くブラウザのみで完結（ネイティブアプリ不可、PWA不要）。
スポット説明順路に沿った各スポットの説明（テキスト＋画像）。内容はJSON等のデータと表示を分離し、非エンジニアが差し替え可能にする。（プロトタイプ実装済：サイン01〜11対応）
スタンプラリー園内サインのQR読取ごとに「発見」を記録。収集状態はブラウザ内（localStorage等）。規定数（暫定3か所）到達で花送信を解放。段階的ごほうび（公園写真DL等）に対応できる構造。
花の選択・送信色（5種）・種類（3種）・任意メッセージを選び POST /api/flowers。送信成功後、スマホ画面に自分の花を即座に表示して手応えを返す（サイネージ反映の数分遅れを体験上カバー）。
設定の外出しスポット内容・花の選択肢・ごほうび閾値などは設定ファイル（JSON）にまとめ、非エンジニアが差し替え可能に。

7. データモデル・API 仕様

プロトタイプで確定済みの仕様です。本番でもこの構造を踏襲します（DB製品は本番化に伴い変更可）。

データモデル（flowers テーブル）

カラム型説明
idINTEGER (PK)主キー（自動採番）
colorTEXT花の色：pink / yellow / white / purple / red
kindTEXT花の種類：a（コスモス）/ b（チューリップ）/ c（さくら）
messageTEXT任意のひとこと（空可・最大文字数制限あり）
created_atTEXT送信日時（ISO8601、サーバー付与）

集計：累計＝全件数 / 本日＝created_at が当日のもの。

API エンドポイント

メソッドパス役割
POST/api/flowers花を1件登録。body: {color, kind, message}。created_atはサーバー付与。
GET/api/flowers?since=IDsince以降の花リストと集計（total / today / latest_id）を返す。表示ページが読込時に呼ぶ。since=0で全件。
GET/api/stats累計・本日のみを軽量に返す（補助用）。
POST/api/demo/seedデモ用サンプル投入。本番では無効化（環境変数で制御）。

レスポンス例（GET /api/flowers?since=120）

// HTTP 200
{
  "flowers": [
    { "id": 121, "color": "pink", "kind": "a",
      "message": "きれい", "created_at": "2027-04-10T13:24:05+09:00" }
  ],
  "total": 121,      // 累計
  "today": 34,       // 本日
  "latest_id": 121   // 次回 since に使う
}

8. 非機能要件

項目要件
可用性会期中（数週間）の連続稼働。無人運用前提で、障害時は自動復旧（再起動）すること。
性能・規模想定来場規模・同時参加人数に基づき設計（具体数は要確認・第12章）。ピーク時の投稿集中に耐えること。
対応ブラウザモバイル：iOS Safari / Android Chrome 最新。表示：OTOMO搭載ブラウザ（Chromeベース想定、実機確認要）。
セキュリティCORS限定、入力検証、レート制御、最低限のスパム対策。個人情報を扱わない。
保守性表示内容・選択肢・閾値をデータ（JSON）で外出しし、非エンジニアが更新可能に。
軽量性フロントは素のHTML/CSS/JSを基本とし、重いビルド工程・過剰な依存を避ける。

9. 本番化で必須の追加対応（重点）

🔴 本委託の核心はここです プロトタイプは「動く」状態ですが、会期中の実運用に必要な信頼性・安全性・運用性は未対応です。以下を本番品質で作り込んでください。
課題本番で必要な対応
① データ永続化
必須		プロトタイプは無料ホスティング（Render.com 無料プラン）でSQLiteを使用しており、再起動でデータが消失する。永続ストレージ／マネージドDBへ移行し、会期中のデータを保全する。
② 本番ホスティング
必須		無料プランは15分無アクセスでスリープする。常時稼働する本番ホスティングを選定・構築する（クラウド/VPS等は提案を歓迎）。
③ メッセージ監視
必須		来場者の自由入力が公開表示される。不適切投稿に備え、フィルタ強化＋（事前確認 or 事後非表示）の運用フローと、必要なら簡易管理画面を設計・実装する。
④ CORS・堅牢化
必須		現状は全オリジン許可。本番公開URLに限定。レート制限・基本的な入力堅牢化。
⑤ OTOMO実機検証
必須		実機の表示仕様（リロード方式・周期・ちらつき）を検証し、ポーリング間隔・一斉開花のタイミングを最適化する。
⑥ 監視・復旧
推奨		死活監視、自動再起動、ログ収集。無人運用での障害早期検知。
⑦ 負荷対策
推奨		同時アクセス・投稿集中への耐性。必要に応じキャッシュ・キュー等。

10. 技術スタック

プロトタイプは以下で構築されています。本番化にあたり、より適切な構成があれば提案を歓迎します（特にDB・ホスティング）。ただしフロントの軽量方針は維持してください。

パートプロトタイプ構成本番方針
サーバーFastAPI（Python）+ SQLiteFastAPI継続可。DBは永続化対応へ（PostgreSQL等の提案可）。
表示ページ素のHTML / CSS / JavaScript（SVGアニメ）軽量方針を維持。フレームワーク不要。
モバイルページ素のHTML / CSS / JavaScript同上。
ホスティング静的：GitHub Pages / API：Render.com 無料本番ホスティングを選定（提案歓迎）。
