# 開発指示書：参加型サイネージ「みんなで咲かす未来の街」
> このファイルは Claude Code に最初に読み込ませる設計の出発点です。
> プロジェクトのルートに `SPEC.md` として置き、Claude Code 起動後に
> 「SPEC.md を読んで、まず全体構成を提案して」と指示してください。
---
## 0. このプロジェクトの目的
GREEN×EXPO 2027（横浜花博）の大阪府自治体出展で使う、来場者参加型のWeb体験を開発する。
来場者がスマートフォンで庭園を巡り、最後に「花」を選んで送信すると、ゴール地点の
サイネージ（OTOMO）に映る「花が咲く街」に花が加わり、累計が増えていく。
**まずは本番ではなく、社内・自治体への提示と仕様検証のためのプロトタイプを作る。**
動くものを見せて合意形成し、本番開発・会期中運用の要否は別途判断する。
---
## 1. 全体構成（3つのパート）
```
[来場者のスマホ]            [サーバー]                 [OTOMO サイネージ]
 ツアーページ   --送信-->   軽量API + データ保存  <--読込-- 「花が咲く街」表示ページ
 (説明/スタンプ/花選択)      (花をためる/集計)        (数分ごとに読み込み・一斉開花)
```
開発対象は次の3つ。優先順位もこの順とする。
1. **表示ページ**（`display`）= 「花が咲く街」。最初に作る。単体で見栄えがして提示に使える。
2. **サーバー**（`server`）= 花を受け取って保存し、表示ページに渡すAPI。
3. **スマホ向けツアーページ**（`mobile`）= 来場者が操作する画面。
---
## 2. 最重要の設計前提（必ず守る）
### 2-1. サイネージは「数分ごとの読み込み」で更新される
OTOMO の Web 表示機能は、ページをリアルタイムではなく**数分に一度読み込みに行く**仕様。
これを弱点ではなく演出として扱う：
- 表示ページは、**読み込まれた時点での最新状態**をサーバーから受け取って描画する。
- 前回読み込み以降に増えた花を、**まとめて「ふわっと一斉に咲く」アニメーション**で見せる。
- 「数分ごとに、みんなの花が一斉に咲く瞬間」というリズムのある演出にする。
- **WebSocket やリアルタイム双方向通信は使わない。** 表示ページはステートレスに、
  読み込み時にAPIから現在の全花データ（または直近の差分）を取得する方式にする。
### 2-2. 個人の即時の手応えはスマホ側で返す
花を送信した瞬間、来場者**自身のスマホ画面**に「あなたの花が咲きました」と表示する。
サイネージ側の数分の遅れは、これにより体験上問題にならない。
### 2-3. アプリのインストールは絶対に不要
スマホ側はQRから開くブラウザだけで完結するWebページ（PWA不要、ネイティブアプリ不可）。
### 2-4. 精度より体験
位置判定（GPS）は使わない。来訪確認が要る箇所は、現地のQR読み取りで明示的に行う。
---
## 3. 技術スタックの希望
開発者は FastAPI / SQLite の利用経験があり、Windows 環境（Git Bash 設定済み）で開発する。
特別な理由がなければ、馴染みのある構成を優先する。
| パート | 推奨スタック | 備考 |
|--------|--------------|------|
| サーバー | **FastAPI + SQLite** | 経験のある構成。軽量で十分 |
| 表示ページ | 素の HTML + CSS + JavaScript | フレームワーク不要。アニメーションは CSS/Canvas |
| スマホページ | 素の HTML + CSS + JavaScript | 同上。軽さ最優先 |
- ビルド工程は最小に。プロトタイプ段階では複雑なバンドラを入れない。
- 依存は最小限に保つ。追加する場合は理由を説明してから入れること。
---
## 4. データモデル（最小）
SQLite に「咲いた花」を1件ずつ記録する。最小のスキーマ案：
```
flowers
  id          INTEGER  主キー
  color       TEXT     花の色（例: pink / yellow / white / purple / red）
  kind        TEXT     花の種類（例: a / b / c）
  message     TEXT     任意のひとことメッセージ（空でも可）
  created_at  TEXT     送信日時（ISO8601）
```
集計はこのテーブルから算出する：
- 累計＝全件数
- 本日＝created_at が当日のもの件数
---
## 5. APIエンドポイント（最小）
| メソッド | パス | 役割 |
|----------|------|------|
| POST | `/api/flowers` | 花を1件登録。body: `{color, kind, message}`。サーバー側で created_at を付与 |
| GET  | `/api/flowers?since=ID` | since 以降の花リストと、累計・本日の集計を返す。表示ページが読込時に呼ぶ |
| GET  | `/api/stats` | 累計・本日のみ軽量に返す（カウンタ表示の補助用、任意） |
- `since` は表示ページが「前回どこまで描いたか」を渡すための仕組み。
  これにより、表示ページは差分だけ受け取って「新しい花」を一斉開花させられる。
- レスポンスは JSON。CORS は開発中は緩めでよいが、本番前に要確認とコメントを残す。
### 投稿内容のチェック
- message には NGワードフィルタ（簡易でよい）と最大文字数制限を入れる。
- 本番では表示前チェックの仕組みが要る旨を、コード内コメントで明記する。
---
## 6. 表示ページ（display）の詳細
**これを最初に作る。** サーバーが未完成でも、ダミーデータで動くようにしておくこと。
- 縦長（43インチ縦型サイネージ）を想定。`viewBox` / レイアウトは縦向き基準。
- 画面下部に街のスカイライン（シルエットでよい）。上に向かって空。
- 花は、街並みの隙間や空に少しずつ増えていく。座標は重なりすぎないよう分散配置。
- 上部または下部に「本日 ◯◯ 輪 / 累計 ◯◯ 輪」を常時表示。
- **一斉開花アニメーション**：読み込み時、APIから新規分を受け取り、それらを
  0.1〜0.3秒ずつずらして「ふわっと咲く」（透明度＋わずかな拡大）。
- ネットワーク断やAPIエラー時も、**エラー画面を全面に出さない**。
  前回の表示を保持し、静かに再試行する（屋外・無人運用のため）。
- 動作確認用に、キー操作などで花を手動追加できるデバッグ機能を付けてよい
  （本番では無効化する前提でフラグ管理）。
### 注意：このプロジェクト本番ではブラウザストレージに依存しすぎない
表示ページの「前回どこまで描いたか」はメモリ上の変数で持つ。
OTOMO 側がページを完全リロードする場合、状態はリセットされ得る。
その場合でも「現在の全花を描く」フォールバックで破綻しないようにする。
---
## 7. スマホ向けツアーページ（mobile）の詳細
1ページ内（または軽いページ遷移）で次を完結させる：
- **入口**：QRから開くトップ。ツアー開始。
- **園内スポット説明**：順路に沿って、各スポットの説明（テキスト＋画像）。
  内容は後から差し込めるよう、データ（JSON等）と表示を分離しておく。
- **スタンプラリー**：園内サインのQRを読むたびに「発見」を記録。
  収集状態はブラウザ内（localStorage 等）に保持。規定数で公園写真をダウンロード可能に。
  - 段階的ごほうび（例：3個でA、5個で全部）に対応できる作りにする。
- **花を選んで送信**：色・種類・任意メッセージを選び、POST /api/flowers を呼ぶ。
  送信成功後、**スマホ画面に自分の花を即座に表示**して手応えを返す。
スポット説明や花の選択肢、ごほうび閾値などは**設定ファイル（JSON）にまとめ**、
非エンジニアでも内容を差し替えられる構造にする。
---
## 8. 作業の進め方（Claude Code への依頼順）
1. SPEC.md を読み、リポジトリ構成（display / server / mobile のディレクトリ分割）を提案する。
2. **まず display を、ダミーデータで単体動作するように作る。** ブラウザで開いて確認できる形に。
3. 一斉開花アニメーションと累計表示を仕上げ、縦型レイアウトで見栄えを整える。
4. 次に server（FastAPI + SQLite）を作り、display を API 連携に切り替える。
5. 最後に mobile を作り、送信→display反映（数分想定）の一連を通して確認する。
6. 各段階で、ローカルでの起動方法を README に書く。
各ステップ完了ごとに一旦止まり、動作確認の方法を示してから次へ進むこと。

## 9. やらないこと / 範囲外（プロトタイプ段階）
- 本番のサーバー運用・監視・スケーリング設計（別途・制作会社検討事項）
- OTOMO 実機との接続検証（提供元ファンテック社への確認が前提）
- ミャクミャク等キャラクター画像の同梱（ライセンス管理下で別途用意）
- 公園写真の同梱（大阪府広報写真など権利明確なものを別途用意）
- 個人情報の収集（本企画では不要。集めない設計にする）
---
## 10. 開発者向けメモ（環境）
- Claude Code は Node.js 18+ もしくは純正インストーラーで導入（導入済みなら不要）。
- Windows の場合、Git Bash 連携（CLAUDE_CODE_GIT_BASH_PATH）が設定済みであること。
- FastAPI 起動例： `uvicorn app:app --reload`（詳細は生成される README に従う）。
- display と mobile は静的ファイルなので、ローカルでは簡易HTTPサーバーで開けばよい。

## 採用済みの方向性（プロトタイプ初期判断）
- **デザイン**：手描き温かみ系（ベクター調・水彩のような柔らかさ・手書きフォント）
- **display のポーリング間隔（暫定）**：30秒（本番は OTOMO 側の読込周期に合わせて変更）
- **花の選択肢（暫定）**：色 5種 × 種類 3種 = 15通り
